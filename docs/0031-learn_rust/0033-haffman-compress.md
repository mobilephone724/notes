## Technical Report on Huffman Compression Module

### 1. Algorithm Principle

Huffman coding is a lossless data compression algorithm that assigns variable-length codes to input characters, with shorter codes for more frequent characters. The algorithm builds a binary tree (Huffman tree) where each leaf node represents a character, and the path from root to leaf determines the code (0 for left, 1 for right). The tree is constructed by repeatedly combining the two nodes with the lowest frequencies until one root remains.

In this implementation:
- Frequencies of each byte (0-255) are counted.
- A priority queue (BinaryHeap) is used to build the tree, with lower frequency nodes having higher priority.
- Codes are generated by traversing the tree.
- Compression replaces each byte with its Huffman code, packed into bytes.
- Decompression reads bits and traverses the tree to reconstruct bytes.

The compressed data includes a header with frequencies (256 * 8 bytes) and the bit length (8 bytes), followed by the compressed content.

### 2. Code Design

#### High-Level Design
The module is structured into four main sections:
- **Tree Node and Tree**: `HuffmanTreeNode` and `HuffmanTree` for building the Huffman tree.
- **Compressed Code**: `HuffmanCompressedCode` using `BitVec<u8, Msb0>` for bit-level operations.
- **Bidirectional Map**: `HuffmanMap` with `HashMap`s for byte-to-code and code-to-byte mappings.
- **Public API**: `compress` and `decompress` functions, plus file I/O wrappers.

The design separates concerns: tree building, code generation, mapping, and I/O. It uses Rust's ownership and borrowing for safety, with no unsafe code.

#### Rust-Featured Low-Level Code Design
- **Error Handling**: Custom `HuffmanError` enum with variants for different error types (InvalidInput, TreeBuildError, etc.), implementing `std::error::Error`. This provides type-safe, descriptive errors instead of strings. `FileCompressError` wraps I/O and Huffman errors.
- **Generics and Traits**: `HuffmanTreeNode` derives `Eq`, `PartialEq`, `Ord`, `PartialOrd` for heap ordering. `HuffmanCompressedCode` derives `Clone`, `PartialEq`, `Eq`, `Hash` for use in `HashMap`.
- **Memory Management**: Uses `Box` for tree nodes to avoid stack overflows. `BitVec` for efficient bit manipulation. Vectors and hashmaps for dynamic storage.
- **Iterators**: Custom `iter()` method on `HuffmanCompressedCode` returns `bitvec::slice::Iter`, allowing iteration over bits.
- **Type Safety**: Strong typing with custom types like `HuffmanFrequency` (array of 256 u64s). Functions use `Result<T, HuffmanError>` for error propagation.
- **Performance**: BinaryHeap for O(log n) insertions. HashMap for O(1) lookups. Bit-packing minimizes space.

### 3. Vec and BitVec and Their Iters

- **Vec<u8>**: Used for byte sequences (input/output data, compressed result). Provides dynamic sizing, efficient appends (`extend_from_slice`), and indexing. Iterators: `iter()` for borrowing iteration, `into_iter()` for consuming.
- **BitVec<u8, Msb0>**: From the `bitvec` crate, stores bits in bytes with MSB-first ordering. Used in `HuffmanCompressedCode` for variable-length codes. Supports bit-level operations like `push(bool)`, `clear()`. Iterator: `iter()` returns `bitvec::slice::Iter<'_, u8, Msb0>`, yielding `bool` values for each bit.
- **Iterators Usage**: In compression, `compressed_code.val.iter()` loops over bits to pack into bytes. In decompression, `BitVec::from_element(ch).iter()` extracts bits from a byte. This allows precise bit manipulation without manual bit shifting.

### 4. Complex Statements

- **Tree Building Loop**: `while tree.len() >= 2 { let Some(n1) = tree.pop() else { ... }; ... }` - Uses `if let` pattern matching for safe unwrapping from heap, ensuring tree construction.
- **Bit Packing in Compress**: `if *bit { byte |= 1 << (7 - filled); }` - Packs bits into bytes using MSB0, with `filled` counter for byte boundaries.
- **Decompression Bit Reading**: `for (i, b) in bits.iter().enumerate() { if (i as u64) >= remain { break; } compressed_code.push(*b); if let Some(val) = dic.get(&compressed_code) { ... } }` - Reads bits up to `remain`, accumulates in `compressed_code`, checks dictionary on each bit for prefix matching.
- **Frequency Parsing**: `frequency[i] = u64::from_le_bytes(chunk.try_into().unwrap());` - Assumes valid chunks, uses `try_into()` for array conversion, `from_le_bytes` for endianness.

# Understanding the Custom Error Handling in `huffman_compress.rs`

This document explains the custom error handling system implemented in `huffman_compress.rs`. It's designed for beginners in Rust to understand how to create and use custom error types.

## The Problem: Handling Different Kinds of Errors

In the `file_compress` function, we can encounter at least two different types of errors:
1.  **I/O Errors:** These happen when reading from or writing to a file (e.g., file not found, permission denied). These are of type `std::io::Error`.
2.  **Internal Compression Errors:** Our `compress` function can fail for its own reasons, and it signals this by returning an error of type `String`.

A function in Rust can only return one error type. To handle both `std::io::Error` and `String` from `file_compress`, we need to create a new, unified error type that can represent either kind of error.

## The Solution: A Custom Error Enum

Here is the code for our custom error system. We'll go through it line by line.

### 1. The `FileCompressError` Enum

```rust
#[derive(Debug)]
pub enum FileCompressError {
    Io(std::io::Error),
    Internal(String),
}
```

-   `pub enum FileCompressError`: This declares a new public enumeration (enum) named `FileCompressError`. An enum is a type that can be one of several different variants.
-   `Io(std::io::Error)`: This is the first variant, named `Io`. It holds a `std::io::Error` value. We'll use this variant to wrap any I/O errors that occur.
-   `Internal(String)`: This is the second variant, named `Internal`. It holds a `String`. We'll use this to wrap the errors from our `compress` logic.
-   `#[derive(Debug)]`: This is an attribute that automatically implements the `Debug` trait for our enum. This allows us to print the error for debugging purposes (e.g., using `println!("{:?}", my_error);`).

**Why?** This enum acts as a container. An instance of `FileCompressError` will either be an `Io` error or an `Internal` error, allowing us to have a single, consistent error type for our function.

### 2. Implementing `Display` for Pretty Printing

```rust
impl fmt::Display for FileCompressError {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        match *self {
            FileCompressError::Io(ref err) => write!(f, "IO error: {}", err),
            FileCompressError::Internal(ref err) => write!(f, "Internal error: {}", err),
        }
    }
}
```
-   `impl fmt::Display for FileCompressError`: This block implements the `Display` trait from the `std::fmt` module for our `FileCompressError` enum. The `Display` trait is used for user-facing, formatted output, like when you print an error message to the console.
-   `fn fmt(...)`: This is the required function for the `Display` trait. It defines how a `FileCompressError` value should be converted to a string.
-   `match *self`: We use a `match` statement to check which variant of the enum we have.
-   `FileCompressError::Io(ref err) => ...`: If it's an `Io` variant, we get a reference (`ref`) to the inner `std::io::Error` and format it with a "IO error: " prefix. The `{}` in the `write!` macro will use the `Display` implementation of `std::io::Error`.
-   `FileCompressError::Internal(ref err) => ...`: Similarly, if it's an `Internal` variant, we format the inner `String` with an "Internal error: " prefix.

**Why?** This allows us to easily print our custom error in a human-readable way (e.g., `println!("{}", my_error);`). It's a standard practice for all error types.

### 3. Implementing the `Error` Trait

```rust
impl Error for FileCompressError {
    fn source(&self) -> Option<&(dyn Error + 'static)> {
        match *self {
            FileCompressError::Io(ref err) => Some(err),
            FileCompressError::Internal(_) => None,
        }
    }
}
```
-   `impl Error for FileCompressError`: This implements the standard `Error` trait from `std::error`. This is the most important trait for error types in Rust. It marks our type as a standard error, allowing it to interoperate with other libraries and tools.
-   `fn source(&self) -> ...`: This is an optional method in the `Error` trait. It should return the "source" or "cause" of the error, if there is one. This is useful for creating a chain of errors.
-   `FileCompressError::Io(ref err) => Some(err)`: For our `Io` variant, the underlying cause is the original `std::io::Error`. So we return it. This allows someone to inspect the chain of errors.
-   `FileCompressError::Internal(_) => None`: Our `Internal` variant holds a `String`. Since `String` itself doesn't implement the `Error` trait, it can't be an error source. So, we return `None`.

**Why?** Implementing `Error` is the idiomatic way to create an error type in Rust. It signals to other parts of the Rust ecosystem that our type represents an error and allows for things like error chaining.

### 4. Automatic Conversion with `From`

```rust
impl From<std::io::Error> for FileCompressError {
    fn from(err: std::io::Error) -> Self {
        FileCompressError::Io(err)
    }
}

impl From<String> for FileCompressError {
    fn from(err: String) -> Self {
        FileCompressError::Internal(err)
    }
}
```

-   `impl From<T> for U`: These blocks implement the `From` trait. This trait provides a way to convert a value of one type into another.
-   The first block says: "Here is how you convert (`From`) a `std::io::Error` into (`for`) a `FileCompressError`." The implementation simply wraps the `io::Error` in our `Io` variant.
-   The second block does the same for `String`, wrapping it in our `Internal` variant.

**Why?** This is the magic that makes the `?` operator work seamlessly. When you use `?` on a `Result` that might contain a `std::io::Error`, Rust sees that the function needs to return a `FileCompressError`. It then looks for an `impl From<std::io::Error> for FileCompressError` to perform the conversion automatically. The same happens for the `String` error type. This keeps our code clean and avoids manual `match` statements for error handling.

### 5. The Updated `file_compress` Function

```rust
pub fn file_compress(
    file_in_path: &str,
    file_out_path: &str,
) -> Result<(), FileCompressError> {
    let mut file_in = OpenOptions::new().read(true).open(file_in_path)?;

    let mut contents: Vec<u8> = Vec::new();
    file_in.read_to_end(&mut contents)?;

    let compressed = compress(&contents)?;

    let mut file_out = File::create(file_out_path)?;
    file_out.write_all(&compressed)?;

    Ok(())
}
```
-   `-> Result<(), FileCompressError>`: The function's return type is now a `Result` where the error variant is our new `FileCompressError`.
-   `...open(file_in_path)?`: `open` returns a `Result<File, std::io::Error>`. If it's an error, the `?` operator will automatically convert the `std::io::Error` into a `FileCompressError::Io` (using our `From` implementation) and return it from the function.
-   `...read_to_end(&mut contents)?`: Same as above.
-   `compress(&contents)?`: `compress` returns a `Result<Vec<u8>, String>`. If it's an error, `?` will convert the `String` into a `FileCompressError::Internal` (using our other `From` implementation) and return from the function.

**Why?** The function is now much cleaner. We've removed the manual error mapping (`.map_err(...)`) and can use `?` on any function that returns a `Result` with an error type that can be converted into our `FileCompressError`. This makes the code more readable and maintainable.
